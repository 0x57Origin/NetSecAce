Lesson 4.1 ‚Äî The AAA Framework

AAA is a fundamental security framework. Every time you log into a system, all three components are at work.

ü™™

Authentication

Proving and verifying who you are. The system checks your claimed identity against stored credentials.

‚Üí Username + password
‚Üí Fingerprint scan
‚Üí Smart card + PIN

Authorization

Determining what you're allowed to do after your identity is verified. Happens AFTER authentication.

‚Üí Can this user read file X?
‚Üí Can this user delete records?
‚Üí Can this user access admin panel?
üìã

Accounting

Tracking and logging what users do after they're authenticated and authorized. Creates an audit trail.

‚Üí Login timestamp
‚Üí Files accessed
‚Üí Changes made

Real-World Example: Corporate Login

Authentication Sarah enters her username "sarah.jones" and password, then approves an MFA push notification ‚Üí Identity verified
Authorization System checks Sarah's role: "HR Manager" ‚Üí She can access HR portal but NOT the finance system ‚Üí Access granted to HR, denied to finance
Accounting Log entry created: "sarah.jones logged in at 09:14 AM, accessed employee records for John Doe, exported report at 09:32 AM"

üìå Common Exam Mistake

Authentication and Authorization are often confused. Remember: you can be authenticated (logged in) but still not authorized (permitted) to access a specific resource. They are separate steps.

üìå Salting

A salt is a random value added to each password before hashing. Even if two users have the same password, their stored hashes are completely different. This defeats rainbow table attacks.

Lesson 4.4 ‚Äî Core Security Principles

These principles guide how security professionals design and implement defenses. Expect multiple questions about these on the exam.

Least Privilege

Grant users and systems only the minimum access rights needed to perform their job ‚Äî nothing more.

Example: A data entry clerk doesn't need administrator access. A contractor doesn't need access to all company files. Limit the blast radius if an account is compromised.
Need to Know

Access to information is granted only if the job function requires it, even if the person has the correct security clearance level.

Example: An FBI agent with Top Secret clearance still can't access classified nuclear weapon designs unless their assignment requires it.
Separation of Duties

Split critical tasks between multiple people so no single person can commit fraud or make errors without another person being involved.

Example: Person A creates a payment request ‚Üí Person B approves it ‚Üí Person C processes it. One person cannot do all three steps.
Defense in Depth

Use multiple overlapping layers of security. If one layer fails, others continue to protect. Never rely on a single control.

Physical ‚Üí Firewall ‚Üí IDS/IPS ‚Üí Antivirus ‚Üí Application ‚Üí Data Encryption ‚Üí Access Controls
Zero Trust

"Never trust, always verify" ‚Äî assume every user, device, and connection is potentially hostile, even inside the network perimeter.

Traditional model: Trust inside the firewall. Zero Trust: Verify everything, every time, regardless of location. Used by modern cloud architectures.
Fail Secure (Fail Safe)

When a system fails, it should default to a secure state ‚Äî denying access rather than granting it.

Example: If a firewall crashes, it should block ALL traffic (fail secure) rather than allow all traffic (fail open). Exception: physical safety systems (emergency exits should fail open).
Job Rotation

Regularly rotating employees through different roles to prevent long-term fraud and ensure no single person has permanent control over critical functions.

Also helps with cross-training and knowledge sharing. Related: mandatory vacation policy ‚Äî forces someone else to cover, exposing potential fraud.

Lesson 4.5 ‚Äî Access Control Models

Access control models define how access to resources is granted and managed. Each model has different use cases.

DAC ‚Äî Discretionary Access Control Owner-controlled

The resource owner decides who gets access. Most flexible but also most vulnerable if an account is compromised.

Example: Windows file permissions ‚Äî you own a file and can grant read/write/execute to specific users
‚ö†Ô∏è Risk: Users can inadvertently share data with unauthorized parties
MAC ‚Äî Mandatory Access Control System-enforced

The system enforces access based on classification labels. Users cannot change permissions ‚Äî the security policy is set by the administrator. Most restrictive.

Example: Military data classification ‚Äî Unclassified, Confidential, Secret, Top Secret. A Secret-cleared user cannot access Top Secret data.
‚úÖ Strongest model ‚Äî used in government/military
RBAC ‚Äî Role-Based Access Control Role-controlled

Access is granted based on the user's job role. Assign roles to users; roles have permissions attached. Most common in business environments.

Example: "HR Manager" role ‚Üí access to HR portal. "Developer" role ‚Üí access to code repos. "Admin" role ‚Üí access to everything.
‚úÖ Easy to manage at scale ‚Äî update the role, not individual users
ABAC ‚Äî Attribute-Based Access Control Attribute-controlled

Access is granted based on multiple attributes of the user, resource, and environment. Most granular and flexible.

Example: "Grant access IF user's department = HR AND clearance = Secret AND time = business hours AND location = company VPN"
‚úÖ Most flexible ‚Äî used in cloud and zero trust environments
